# Codex Review

## Strengths (implementation-feasibility lens)
- `ARCHITECTURE.md §6` keeps MVP as a single Node 20+ process with async adapter fan-out and no daemon. That is the right cut for Phase 0–1; it avoids IPC, background lifecycle, and install complexity.
- `PLATFORM_LAYER.md §1, §16` is unusually concrete about enforcing the PAL via lint rules. That matters because Windows support will otherwise rot immediately.
- `ADAPTER_SPEC.md §3, §13` defines a credible TS SDK surface and contract-test suite. The split between orchestrator-owned tool execution and adapter-owned provider translation in `§6` is a good boundary.
- `ARCHITECTURE.md §9` choosing SQLite with a repository pattern is realistic for local-first MVP. `better-sqlite3` + WAL is pragmatic for single-user, mostly-read workloads.
- `SAFETY_MODEL.md §4-§7` is implementation-oriented, not aspirational. Scoped approvals, explicit action kinds, and blobbed audit payloads are all buildable.
- `MVP_ROADMAP.md §3` sequences the work mostly correctly: contracts first, one adapter vertical slice before multi-agent debate.

## Risks (specific, with file:section reference)
- `ARCHITECTURE.md §12` and `MVP_ROADMAP.md §178` propose loading user workflows/protocols from `.manthan/*.ts` at startup. In Node this means either bundling/transpiling untrusted TS, shipping a runtime loader (`tsx`, `jiti`, `esbuild-register`), or forcing precompiled JS. None is specified, and this will bite immediately on Windows + ESM.
- `ARCHITECTURE.md §9` picks `better-sqlite3`, while `PLATFORM_LAYER.md §14` rejects native modules without prebuilts. `better-sqlite3` is native. Prebuild availability is usually fine, but it directly conflicts with the “pure JS or prebuilt everywhere” plugin/install story and is a common CI/install failure point.
- `ARCHITECTURE.md §9` says “no raw SQL outside repository,” but the actual schema is JSON-heavy (`metadata_json`, `payload_json`, `layers_json`). Without explicit indexed query patterns, Phase 1 memory queries will devolve into table scans and ad hoc JSON parsing.
- `PLATFORM_LAYER.md §3-§12` over-specifies PAL for Phase 0. `ShellAdapter`, `WatchOps`, ACL intent recording, UTF-8 console handling, named pipes, long-path verification, and sandbox hooks are too much before `manthan init` exists. The seam is right; the initial scope is not.
- `ADAPTER_SPEC.md §3` mixes transport-level content parts, orchestration concerns, pricing, and provider payload retention into one contract. The result is likely fragile versioning: small changes to tool events or usage accounting will trigger SDK major bumps too often.
- `ADAPTER_SPEC.md §13` requires contract tests for cancellation within 500ms and “budget guardrail” rejection before network call. That is good in principle, but several providers/SDKs do not abort promptly or expose deterministic fixture points. The acceptance bar may be unrealistic for external adapter authors.
- `SAFETY_MODEL.md §11-§12.2` understates the risk of in-process plugins. Lint rules do not protect third-party published JS at runtime. If plugins run in-process in MVP, “capability manifests” are advisory, not enforceable.
- `DEBATE_PROTOCOL.md §7-§8` requires replayable, content-addressed transcripts and byte-identical replay outcomes in `MVP_ROADMAP.md §122`. Cross-provider request normalization, prompt templates, and JSON serialization stability are underspecified; this is harder than the docs imply.

## Missing (what's not specified that you'd need to write code)
- Package layout and dependency graph: what lives in `packages/core` vs `packages/orchestrator` vs CLI, and which packages are allowed to depend on which.
- Config format details beyond mentions of `.manthan/config.yaml`: schema, inheritance, env interpolation rules, secrets references, and how per-workspace overrides merge.
- Concrete plugin discovery algorithm: npm global vs local install, `node_modules` scanning vs explicit config paths, resolution precedence, and cache/lockfile behavior.
- SQLite migration format and startup semantics: version table, transactional guarantees, downgrade policy, corruption handling, WAL checkpoint behavior.
- Context packer inputs/ordering: exact bundle schema, token estimation library, per-model tokenizer source, cache key structure beyond examples.
- Git abstraction surface: whether to use `simple-git` or raw `spawn`, and what the typed API actually is.
- Logging/diagnostics contract: pino? debug namespaces? file locations? redaction policy for structured logs separate from audit blobs.
- Test ergonomics: fixture format for recorded provider interactions, how golden transcripts are updated, and whether integration tests run against temp repos or snapshots.

## Must-change before Phase 0 (blocking)
1. Decide the module/runtime story for user-extensible `.ts` files: either “MVP loads only `.js/.mjs`” or “ship `tsx` as the runtime loader.” Do not leave this implicit.
2. Cut PAL v0 down to the minimum needed for Phase 0–1: path ops, process spawn, atomic write, user-data dirs, terminal interactivity. Defer watch/signal/pipe/sandbox surfaces.
3. Specify the monorepo package graph and build toolchain: `pnpm`, `tsup`/`unbuild`/plain `tsc`, ESM policy, test runner, and how CLI binaries are emitted.
4. Freeze the config schema and loading rules, including secrets references and per-workspace config resolution.
5. Define the initial SQLite schema with actual indexes and query shapes for Phase 1 tables, not just indicative tables.
6. Make plugin loading explicit for MVP: first-party adapters only by direct package dependency, or explicit path-based loading. Do not promise general npm discovery yet.
7. Relax/clarify adapter contract tests so third-party adapters can realistically pass them with recorded fixtures.

## Nice-to-have (non-blocking)
1. Replace or justify `better-sqlite3` explicitly against install friction; if kept, document prebuild expectations and fallback behavior.
2. Separate SDK wire types from orchestrator helper types to reduce major-version churn.
3. Add an ADR on ESM/CJS policy and Node package exports before any package is published.
4. Define one golden end-to-end demo repo for CI so Windows/macOS/Linux behavior is exercised consistently.
5. Add a minimal trust model disclaimer to plugin docs: MVP capability manifests are informational until isolation exists.

## Confidence: 0.88

## Top 5 Questions You'd Ask the Author
1. What is the exact Node module strategy for core and plugins: pure ESM, dual package, or CJS-compatible?
2. How do you want user-defined workflows/protocols loaded in MVP without making users install a TS runtime?
3. What is the explicit plugin discovery path in Phase 4: config-listed modules, local workspace dependencies, or npm-global scanning?
4. Which concrete SQLite queries must be fast in Phase 1, and what indexes are required to support them?
5. Are third-party adapters truly a near-term goal, or should MVP formally support only bundled first-party adapters until isolation and loader semantics are real?
