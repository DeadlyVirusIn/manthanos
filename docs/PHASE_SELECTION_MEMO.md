# ManthanOS — Phase Selection Memo

> Bounded strategic evaluation, post-stabilization. Evaluates four
> candidate paths against the evidence record. Not a roadmap. Reads
> as an internal research governance document deciding whether the
> project deserves another phase, and which.
> Date: 2026-05-16.

This memo is constrained by the artifacts that closed stabilization:
`TRUTH_CHECKPOINT.md`, `STABILIZATION.md`, `STABILIZATION_VERDICT.md`,
`STABILIZATION_LESSONS.md`. Every claim below either cites surviving
evidence from those artifacts or is named as missing. No new
evidence is generated by this memo; no roadmap follows from it; no
phase begins from it. It evaluates and ranks.

The four paths under evaluation:

- **Path A** — Narrow single-provider continuity tooling
- **Path B** — Re-open cross-model continuity research
- **Path C** — Pivot toward audit/compliance infrastructure
- **Path D** — Freeze as a research artifact and stop active development

---

## Path A — Narrow single-provider continuity tooling

### 1. Core thesis

Trust-gated re-injection of human-promoted commitments materially
reduces architectural drift in repeat-use AI-assisted engineering
within a single provider. ManthanOS ships as the CLI tooling that
makes this loop usable at the daily-engineer scale for a project
that uses one model.

### 2. Evidence currently supporting it

- **Phase 1.7 A/B (2026-05-15)** — one model, two related plans,
  three trusted facts. Measurable drift reduction. TRUTH_CHECKPOINT
  §1.2, MEDIUM confidence, very small N.
- **Substrate runs end-to-end on Linux** — TRUTH_CHECKPOINT §1.1,
  HIGH confidence.
- **Trust mechanics function as designed** — TRUTH_CHECKPOINT §1.5;
  Step 1 of stabilization confirmed via three new test groups.
- **Hygiene primitives execute deterministically** — TRUTH_CHECKPOINT
  §1.5 (qualitative shape preserved post-decay-fix; magnitudes
  reinterpreted per PHASE2_THEORY appendix).
- **Adaptive shaping reports every omission with a reason** — Phase
  2 deliverable; mechanically verified.

That is the entire supporting record. Phase 1.7 is the load-bearing
evidence; everything else is substrate.

### 3. Evidence missing

- **Non-tautological CpT.** Phase 3 was designed with an
  author-curated corpus and author-written briefs; three reviewers
  flagged this as tautological. A re-designed Phase 3 (corpus the
  operator did not curate, briefs the operator did not author,
  blinded rubric) has not run.
- **Real user behavior on the trust queue.** The promotion UX is
  well-implemented inside a session; whether engineers actually
  initiate the session has zero evidence (Sonnet's <20% estimate
  is a hypothesis, not a measurement).
- **Long-horizon results under corrected decay semantics.** The
  long-horizon JSONLs were generated with the `last_corroborated`
  bug; magnitudes are workload-and-bug-specific. A re-run with
  corrected semantics has not happened.
- **Generalization beyond architectural-continuity tasks.** Phase
  1.7's effect was measured on a continuation brief that explicitly
  depended on the prior plan. Whether the effect persists on
  debugging, refactoring, code review, or migration tasks is
  unknown.

### 4. Primary risks

- **Technical**: Adaptive shaping at tight budgets may drop facts
  that turn out to be load-bearing for a real plan; current
  implementation has per-omission reporting but no semantic
  preservation. Substrate self-bounding under realistic neglect
  (attention ≤ 0.05) has not been simulated.
- **Behavioral**: The trust queue requires sustained operator
  attention with no initiation hook. Sonnet's review estimated
  <20% sustained adoption; this is the median engineering-tools
  base rate, not pessimism.
- **Adoption**: 19-command CLI surface, UUID-based fact handles,
  first-session-empty-brain produces zero visible value. The
  empirical first-hour user has not been observed.
- **Competitive**: Anthropic and Cursor each own the distribution
  layer (IDE / model API) and could ship "Project memory with
  decision log" in 6–8 weeks. Both have done it before. The
  feature gap is small; the distribution gap is everything.
- **Operational**: Cross-platform parity is asserted but only
  Linux has been operationally exercised. CI smoke tests on macOS
  and Windows do not constitute validation.

### 5. Time-to-signal

- First useful signal (non-tautological Phase 3 result):
  **4–8 weeks** with disciplined experimental design.
- Real-user signal (small beta, 3–10 engineers, 1 month observation):
  **8–16 weeks**.
- Public-release readiness (security hardening per
  STABILIZATION §2.2 deferred list): **3–6 months** if those items
  are reauthorized.

### 6. Failure modes

- **CpT measurement shows healthy ≈ empty.** The mechanism does
  not materially help on tasks outside the engineered Phase 1.7
  microcosm. The thesis is empirically refuted on the only
  question that matters.
- **Anthropic ships Claude Projects with a decision log.** Product
  category is absorbed before adoption signal can compound.
- **Real-user adoption stalls under 5% sustained queue maintenance.**
  Operationally invalid even if mechanically correct.
- **The trust queue's initiation problem is intractable in CLI
  form.** Adoption requires IDE integration; substrate work is the
  prerequisite, not the product. The product never materializes in
  CLI form; the substrate work was the actual deliverable.

### 7. Why this path could still matter

The audit-first design produces a forensic record of AI-assisted
engineering that is unusual in the space. Phase 1.7 is one of the
few small-N controlled demonstrations in this research area;
adding non-tautological Phase 3 results would strengthen a
genuinely under-documented research direction. The substrate is
usable as research infrastructure regardless of product outcome.

### 8. What work becomes legal under Path A

- Non-tautological Phase 3 design and execution.
- Long-horizon re-run under corrected decay semantics, framed
  as substrate-reinterpretation activity per LESSONS §7.1.
- attention=0.05 long-horizon stress simulation.
- IDE-plugin investigation as a separately-scoped prototype, not
  as core product work.
- Public-release prep: security hardening per STABILIZATION §2.2,
  README sharpening, package distribution.
- Solicitation of small beta users (3–10 engineers) once security
  hardening reaches "minimum installable" threshold.

### 9. What remains prohibited under Path A

- Multi-provider orchestration.
- Multi-agent runtime.
- Frontend command center (FUTURE_COMMAND_CENTER.md stays
  quarantined; the doc may be cited as future-vision artifact but
  contributes zero to active work).
- SaaS infrastructure.
- "AI operating system" framing in any user-facing surface.
- Phase 3 reruns in tautological design.
- Marketing copy ahead of evidence.

### 10. Honest probability assessment

| Outcome | Probability |
|---|---|
| Useful research result (citable Phase 3 finding regardless of direction) | ~55% |
| Viable product (sustained user adoption, positive unit economics or comparable signal) | ~15% |
| Technically-interesting dead-end | ~55% |

Probabilities do not sum to 100 — they are not mutually exclusive. A
useful research result is compatible with a dead-end as a product.

## Path B — Re-open cross-model continuity research

### 1. Core thesis

Trusted continuity built against one model materially influences
the output of a second, different model on the same project. The
substrate is provider-portable rather than provider-bound. E6.1
is the bounded falsification experiment that resolves this.

### 2. Evidence currently supporting it

- **Architectural fact**: the `AgentAdapter` contract is
  provider-neutral. The orchestrator and substrate code paths do
  not assume any specific provider.
- **Mechanical fact**: `@manthanos/adapter-openai` was built in
  ~23 minutes and integrated without changes to `adapters-sdk`,
  `orchestrator`, or `plan-runner`. The provider-neutrality is
  not aspirational; it is observed.
- **Indirect fact**: Phase 1.7 demonstrated that re-injection works
  at all. Whether the effect transfers across providers is the
  question, not whether the mechanism works.

That is the entire supporting record for B specifically. The
cross-model thesis itself has zero empirical evidence.

### 3. Evidence missing

- **Any positive cross-model result.** E6 (2026-05-16) refuted
  the minimal-adapter approach. E6.1 was designed to test the
  proper-adapter approach and did not execute (provider billing).
- **More than one secondary model.** Even a clean E6.1 pass
  establishes "demonstrated against one model on one brief"; it
  does not establish portability.
- **More than one brief.** E6.1 uses `auth-reset-password.brief`
  exclusively. A second-brief result is required before any
  positioning claim.
- **Multi-provider operational viability.** Cost ceilings,
  authentication ergonomics, and cross-API failure modes across
  three or more providers are unstudied.

### 4. Primary risks

- **Technical**: OpenAI's strict-mode `response_format` may reject
  the PLAN_TOOL schema for compliance reasons; the adapter halts
  cleanly but execution stops. Provider-specific framing biases
  may make cross-provider portability weaker than the architecture
  implies.
- **Behavioral**: A clean cross-model pass on one brief still
  requires E6.2 (multi-model, multi-brief) before any positioning
  claim. Defending the result against "this is one demo" is hard.
- **Adoption**: Multi-provider operation is operationally heavier
  than single-provider. Engineers' real workflow is usually one
  provider; the cross-model demo may not have user-facing payoff.
- **Competitive**: Vendor-neutral memory layers exist (Letta,
  Mem0, MCP-shaped tools). Even a positive E6.1 doesn't establish
  ManthanOS as the defensible cross-model layer.
- **Operational**: Multi-vendor billing, multi-API auth, multi-CLI
  adapter maintenance — all add operational weight that solo
  development cannot easily carry.

### 5. Time-to-signal

- **E6.1 execution once funded**: <1 day (mechanical path verified).
- **E6.2 multi-model design + execution**: 1–2 weeks.
- **E6.3 multi-brief**: 2–4 weeks.

This is the **cheapest informative path** measured per signal-dollar.
A single ≤$20 OpenAI top-up enables E6.1; the result resolves the
cross-model thesis one way or the other within a week.

### 6. Failure modes

- **E6.1 fails cleanly** (C-healthy ✓, X-healthy ✗). Cross-model
  thesis empirically refuted at the proper-adapter level. The
  refutation is stronger than E6's; Path A is finalized with
  evidence rather than by default.
- **E6.1 succeeds on auth brief, fails on db brief.** The cleanest
  positive result still requires E6.2 to defend; the cleanest
  negative result is unambiguous.
- **E6.1 produces ambiguous data on both briefs.** STABILIZATION
  §6.4 default-to-A applies; we end up where Path A is anyway.
- **OpenAI billing remains unresolved.** Path B is gated on
  external resource the project doesn't control.

### 7. Why this path could still matter

The original pain (cross-model handoff) was real and remains
unsolved. A working cross-model demonstration would be unusual in
the space — most "AI memory" projects assume single-provider use
or rely on text-paste copy mechanisms. Even a single demonstration
shifts the cross-model thesis from "refuted" to "open with one
datum," which is a substantively different epistemic state.

### 8. What work becomes legal under Path B

- E6.1 execution on funded provider or explicitly-authorized
  substitute.
- E6.2 multi-model after E6.1 succeeds (two additional providers
  maximum in the first cycle).
- Adapter hardening for one additional provider (Codex CLI parser
  fix, or direct Gemini API, but only one).
- Cross-model bundle-portability research (does the same bundle
  produce comparable outputs across providers).

### 9. What remains prohibited under Path B

- Multi-agent orchestration or any agent-routing logic.
- Debate engines, swarm patterns, autonomous agent chains.
- FUTURE_COMMAND_CENTER work in any form — it explicitly cannot
  re-enter active scope under Path B's narrowed cross-model
  framing.
- Restoring the "underneath whichever AI you happen to use" framing
  before E6.2 has produced multi-model evidence.
- Marketing copy referencing cross-model portability while only
  E6.1 has executed.

### 10. Honest probability assessment

| Outcome | Probability |
|---|---|
| Useful research result (E6.1 produces clean signal in either direction) | ~70% |
| Viable product (sustained user adoption directly attributable to cross-model functionality) | ~10% |
| Technically-interesting dead-end | ~50% |

The research-result probability is higher than Path A's because
E6.1's binary nature makes it nearly impossible to produce
uninformative output; either it works or it doesn't, and either is
citable.

## Path C — Pivot toward audit/compliance infrastructure

### 1. Core thesis

The audit substrate is the actual differentiator. ManthanOS
repositions as auditable decision history for AI-assisted
engineering in regulated industries (financial services, healthcare,
defense, public-sector AI procurement) where provenance-tagged
record of AI-assisted decisions has legal/compliance value, not
productivity value.

### 2. Evidence currently supporting it

- **Hash-chained audit log functions** (TRUTH_CHECKPOINT §1.4,
  HIGH confidence for accidental-corruption detection).
- **Provenance + tier rendered into every prompt** (Phase 1 ship,
  verified in code).
- **Trust-event audit semantics are clean post-Step 1 fixes**.
- **Architecture is regulated-industry-grade in shape** — Codex's
  review explicitly: *"regulated-industry-grade, far more than 'AI
  re-states architectural decisions correctly' requires"*.
- **Regulatory tailwinds**: EU AI Act, NIST AI RMF, sector-specific
  AI governance frameworks. These are external facts the project
  did not create but can align with.

### 3. Evidence missing

- **Any regulated-industry customer interview.** Zero. Not one.
- **Any pilot or letter of interest.** Zero.
- **Any regulatory framing document** mapping ManthanOS audit
  features to specific compliance frameworks (SOC2, HIPAA,
  FedRAMP, EU AI Act sub-clauses).
- **External audit anchoring** (transparency log or signed
  external checkpoint) — required for actual compliance use; not
  implemented.
- **Adapter sandboxing** — required for security posture in a
  regulated environment; not implemented.
- **OS-keychain integration** — required for any enterprise
  install; not implemented.

### 4. Primary risks

- **Technical**: External anchoring + sandboxing + key management +
  cross-platform behavioral testing = significantly larger
  hardening scope than the stabilization deferred list anticipated.
  Time investment is months, not weeks.
- **Behavioral**: Regulated-industry sales cycles are 6–18 months
  per deal. The project cannot afford to build for two years
  before signal.
- **Adoption**: Regulated buyers do not adopt research-grade
  prototypes. The current README's "research-grade prototype"
  framing is correct and disqualifying for this path
  simultaneously.
- **Competitive**: Splunk, Datadog, AWS CloudTrail-flavored
  audit-observability tools own this space. Adding "AI events" to
  any existing audit pipeline absorbs the use case without giving
  ManthanOS a foothold.
- **Operational**: Founder skill set, as observed in the codebase,
  is substrate-builder. Enterprise-buyer pitch, regulatory
  framing, compliance-pilot navigation are different skill stacks.
  Solo execution of this path is unlikely.

### 5. Time-to-signal

- First compliance-pilot conversation: **30 days** (feasible
  with customer-discovery effort).
- Pilot deployment: **6–12 months**.
- SOC2 / regulatory-certification readiness: **1–2 years**.
- First defensible commercial signal: **12–18 months**.

This is the **longest time-to-signal path by a factor of 3–10×**
relative to A or B.

### 6. Failure modes

- **No compliance buyer cares about AI-engineering-specific audit**
  (separate from generic AI-output audit). The market segment
  doesn't exist at the granularity ManthanOS occupies.
- **First pilot fails on security gaps** (in-process adapters,
  plaintext keys). Pilot is lost; the security cost-of-entry was
  not paid before customer signal.
- **Incumbent tools absorb the use case** by adding "AI provenance"
  features to existing audit pipelines. ManthanOS has no
  distribution advantage in this market.
- **The pivot reveals an unbridgeable founder-skill gap**.
  Substrate-builder cannot reach enterprise-buyer in solo mode
  within a useful timeframe.

### 7. Why this path could still matter

The audit substrate IS regulated-industry-grade in shape. The
regulatory tailwinds are real. "Provably-audited AI-assisted
decision history" is a genuine product space — just not the one
ManthanOS was built for. This is the **only one of the four paths
with a substantial moat**: compliance-bound buyers do not switch
lightly, and the audit-first design defends a category most
competitors cannot enter without re-architecting.

### 8. What work becomes legal under Path C

- Customer-discovery interviews (5–10 minimum before any code
  work).
- Regulatory framing document mapping audit features to compliance
  framework clauses.
- External audit anchoring design and prototype.
- Adapter sandboxing design and prototype.
- OS-keychain integration.
- Enterprise-buyer landing page.

### 9. What remains prohibited under Path C

- Productivity-tool framing in any user-facing surface.
- Solo-engineer positioning.
- Cross-model continuity work (orthogonal to this path's value
  proposition).
- FUTURE_COMMAND_CENTER work (the multi-agent vision is a
  productivity story, not an audit story; explicitly prohibited
  under C).
- Any code work that requires the substrate to be production-grade
  before customer signal exists.
- Generic AI-engineering audit framing (must be sector-specific
  from day one).

### 10. Honest probability assessment

| Outcome | Probability |
|---|---|
| Useful research result (the path produces business artifacts, not research artifacts; conversion to research finding is low) | ~25% |
| Viable product (sustained customer pipeline; SOC2 trajectory) | ~25% |
| Technically-interesting dead-end | ~30% |

P(viable product) is the highest of the four paths *if* the
founder-skill mismatch resolves (via partnership, hire, or
acquired expertise). In solo-execution mode the probability drops
sharply.

## Path D — Freeze as a research artifact and stop active development

### 1. Core thesis

The project has reached a stable, honest, well-documented point.
The four stabilization artifacts and the Phase 1.7 demonstration
constitute the entire publishable record. Continuing produces
diminishing returns; freezing preserves the substrate and lessons
for future researchers without further mechanism accumulation.

### 2. Evidence currently supporting it

- **The four stabilization artifacts are publishable as-is**
  (TRUTH_CHECKPOINT, STABILIZATION, STABILIZATION_VERDICT,
  STABILIZATION_LESSONS). They constitute a methodology
  contribution that is genuinely unusual in the space.
- **The substrate is honestly scoped and forkable** (BSL 1.1 today,
  Apache 2.0 at conversion date).
- **Phase 1.7 is a preserved, citeable demonstration**.

### 3. Evidence missing

- **Whether the lessons would actually transfer** to other teams
  without an active maintainer instantiating them.
- **Whether the substrate becomes valuable as a research artifact**
  specifically vs. just as code other people could fork.
- **Whether freezing is a soft form of abandonment** that disguises
  itself as a deliberate decision.

### 4. Primary risks

- **Technical**: Bit-rot. Dependencies decay. better-sqlite3,
  openai SDK, claude SDK, Node version compatibility — each
  becomes a maintenance burden the frozen project cannot pay.
- **Behavioral**: Research artifacts get cited but not used unless
  actively curated.
- **Adoption**: Frozen artifacts have asymptotic adoption near
  zero unless they enter a citation network. ManthanOS is not in
  one.
- **Competitive**: Even as a research artifact, the AI-engineering
  audit space will move; the artifact gets superseded within
  12–24 months.
- **Operational**: Zero. There is no operational risk to freezing,
  which is also its tell that the path is comfortable rather than
  decisive.

### 5. Time-to-signal

- **Immediate**: freeze + publish + close.
- **Useful-citation signal**: 6–18 months (academic-software
  publication cycle).
- **Useful-product signal**: probably never; this path is
  definitionally not a product path.

### 6. Failure modes

- **The artifact is ignored entirely** (most likely outcome).
- **The artifact is misused** — cited as evidence for claims it
  does not make, or forked into derivative products that ignore
  the stabilization discipline.
- **Lessons are not transferable without active project**. The
  lessons document is self-contained, but practitioners
  typically learn methodology from active examples, not frozen
  ones.
- **The decision to freeze is itself a soft form of abandonment**
  dressed as a deliberate choice. This is the failure mode the
  project's own discipline culture is most blind to.

### 7. Why this path could still matter

The lessons document is a genuine public good. The discipline of
publicly closing a project with a truth-checkpoint, a constitution,
a verdict, and a lessons artifact is rare. The substrate is
forkable and the four-document arc is reusable as a template for
other research-grade prototypes. This is a non-zero contribution
even with zero further development.

### 8. What work becomes legal under Path D

- Final public-release prep (README finalization, CHANGELOG,
  archive notice).
- Academic-software publication of the lessons document.
- Migration of any open work items to a frozen-status.
- Transfer of trademark / license metadata if needed.

### 9. What remains prohibited under Path D

- Any new mechanism work.
- Any positioning changes after the public release.
- "We might un-freeze later" hedging language. Either freeze means
  freeze, or it isn't a real decision.
- New phase plans, new theory documents, new experimental designs.
- Active maintenance beyond security patches to dependencies.

### 10. Honest probability assessment

| Outcome | Probability |
|---|---|
| Useful research result (lessons doc cited; substrate forked or referenced) | ~40% |
| Viable product (this path is definitionally not a product path) | 0% |
| Technically-interesting dead-end (the defined outcome of this path; not a failure mode) | ~80% |

Path D's high "dead-end" probability is not a failure; it is what
the path explicitly chooses. The 40% useful-research probability
reflects realistic citation outcomes for a small, niche,
research-grade artifact in a rapidly-moving space.

## 11. Comparative ranking

### Cheapest informative path

1. **B** — E6.1 execution is <1 day once funded; produces a
   binary, citable result either way. The cheapest signal-per-dollar
   in the four paths.
2. **D** — instant; produces no further signal but consumes
   nothing.
3. **A** — 4–8 weeks to first signal; non-trivial but bounded.
4. **C** — 30 days to first conversation, 6–12 months to first
   actual signal. The slowest.

### Strongest evidence alignment

1. **A** — aligns with Phase 1.7 (the strongest piece of
   surviving evidence); requires no new evidence to begin.
2. **D** — aligns with the lessons document (the project's most
   differentiated output); requires no new evidence.
3. **B** — aligns with architectural neutrality but the
   cross-model thesis itself has zero empirical backing.
4. **C** — aligns with audit-substrate quality but the
   compliance-buyer hypothesis has zero empirical backing.

### Highest research value

1. **B** — E6.1's binary outcome is citable in either direction;
   adds to a thin literature on cross-model continuity.
2. **D** — lessons document and methodology pattern are
   genuinely transferable.
3. **A** — non-tautological Phase 3 result is citable but
   incremental relative to Phase 1.7.
4. **C** — produces business artifacts, not research artifacts.

### Highest product potential

1. **C** — only path with a substantial moat (compliance-bound
   buyers); highest variance.
2. **A** — limited but real; competitive overhead is severe.
3. **B** — relevant to product potential only via Path A;
   cross-model is a feature, not a category.
4. **D** — zero.

### Highest risk-adjusted realism

1. **A** — evidence-aligned, founder-skill-matched, citable
   results regardless of product outcome, narrowest scope of work.
2. **B** — realistic if funded; produces clean signal in
   bounded time; resolves a question that Path A leaves open.
3. **D** — realistic but explicitly stops gathering evidence;
   the lessons document is already produced; further freezing
   adds no value to the artifact already created.
4. **C** — highest variance, longest time-to-signal,
   founder-skill mismatch in solo mode.

## 12. Final recommendation

**Path A — Narrow single-provider continuity tooling.**

The recommendation follows strictly from the stabilization
evidence, the operational lessons, and the anti-drift rules.

### From the stabilization evidence

Phase 1.7 is the only piece of surviving empirical evidence with
direct behavioral relevance. It supports single-provider
continuity. It does not support cross-model continuity, compliance
positioning, or freeze-as-publication. Aligning the next phase
with the strongest surviving evidence is the only choice the
evidence record uniquely supports.

### From the operational lessons

LESSONS §7 names three open questions in priority order: Q1
(non-tautological CpT), Q2 (cross-model port), Q3 (queue
maintenance). Q1 is named as the highest-priority question
because it determines whether anything else matters. Q1 is Path
A's required work. Q2 is Path B's required work. Following the
priority order, Q1 — and thus Path A — is the work that must
happen first.

### From the anti-drift rules

STABILIZATION §7 codified twelve anti-drift rules. The relevant
ones for this decision:

- *"No mechanism work without an unresolved product dependency."*
  Path A has the clearest product dependency (does continuity
  help on real work). Path C has no resolved-by-current-work
  dependency. Path D explicitly stops mechanism work.
- *"No future-facing positioning without evidence."* Paths B and
  C both require positioning to enter their first useful work
  cycle; A requires only the existing positioning.
- *"No optionality preservation without bounded follow-through."*
  Path B's empty matrix was the project's last optionality-cost.
  Path A pays no optionality cost.

### What this recommendation explicitly is not

- **Not a refutation of Path B.** E6.1 remains available as a
  bounded side-experiment under Path A's authorization, when
  OpenAI billing resolves or a substitute is explicitly
  authorized. Path B's question is preserved; Path B is not the
  primary phase.
- **Not a refutation of Path C.** Compliance-buyer signal
  collection becomes legal under Path A's §8 ("solicitation of
  small beta users") and can produce the external signal that
  STABILIZATION §6.3 named as the precondition for Path C
  re-evaluation. Path C is preserved as a re-evaluation target,
  not as the current phase.
- **Not a refutation of Path D.** If Path A's non-tautological
  Phase 3 produces a null or negative result on the central
  question, Path D becomes the evidence-supported direction at
  that point. Path D is preserved as a contingent outcome of
  Path A, not as the current phase.

### Required gate before Path A authorization

Before any Path A work begins, the user must explicitly authorize
the new phase. Per STABILIZATION §9 ("That next phase MUST begin
with its own bounded plan analogous to this one before any code
is written"), Path A requires its own constitution document —
analogous to STABILIZATION.md — before any work is performed
under its scope. This memo evaluates; it does not authorize.

### One-line statement

> The strongest single-step move from the current state is to
> answer the Phase 3 question — does continuity help on real
> work — under a non-tautological experimental design. That is
> Path A's defining work. Every other consideration is
> secondary until Q1 has an evidence-backed answer.

before authorization.*
