# Codex Final Hardening Review

## Strongest remaining concern (1 paragraph)
The weakest mechanical seam is the claimed combination of replayability, audit truth, and SQLite-backed local-first persistence. The docs split state across SQLite rows, append-only JSONL, and content-addressed blob files, but never define a single crash-consistent commit protocol for “write blob -> append audit -> commit DB row -> expose to replay” on Windows/macOS/Linux. In Phase 0–1, this will produce heisenbugs: audit events that reference missing blobs, DB rows with no matching audit range, replay runs that cannot reconstruct byte-identical inputs after partial failure, and WAL/rename/AV-lock issues on Windows that violate the “if it happened, it’s in the log” claim. This is more dangerous than ordinary MVP roughness because it corrupts the substrate the rest of the product depends on. Refs: `ARCHITECTURE.md §9-10`, `SAFETY_MODEL.md §7`, `WORKFLOWS_SPEC.md §3, §10-11`, `PLATFORM_LAYER.md §6`.

## What is still underdesigned (5–8 bullets, file:section refs)
- SQLite is justified only as “single-user local-first,” but the real hot path is hybrid: SQLite plus thousands of JSON blobs plus debate transcripts. There is no retention/compaction/snapshot strategy for long-lived replay performance or audit/blob directory growth. Refs: `ARCHITECTURE.md §9`, `SAFETY_MODEL.md §7`, `OBSERVABILITY.md §3.4, §6`.
- Determinism stops at JSON canonicalization, but not at input ordering. File walks, `.gitignore` traversal, SQL tie-break ordering, parallel round event ordering, and provider SDK retries can all change payload hashes without changing user-visible semantics. Refs: `DEBATE_PROTOCOL.md §7.1, §10`, `WORKFLOWS_SPEC.md §6, §11`, `BOOTSTRAP_PROTOCOL.md §6`.
- Plugin architecture is explicitly unsafe for MVP, but the docs still let custom protocols/workflows/plugins shape runtime behavior before isolation exists. In-process ESM can monkey-patch globals, builtins, fetch, timers, or PAL references before the safety model sees anything. Refs: `SAFETY_MODEL.md §11`, `ARCHITECTURE.md §13, §17.12`, `PLATFORM_LAYER.md §14`, `ADAPTER_SPEC.md §5-6`.
- Cost control is too dependent on adapter honesty and pre-call estimates. Real SDK retries, provider-side token accounting drift, tool-call loops, and repeated re-packing are not budgeted as first-class runtime effects. Refs: `ARCHITECTURE.md §11`, `ADAPTER_SPEC.md §8-9`, `WORKFLOWS_SPEC.md §8-9`, `DEBATE_PROTOCOL.md §9`.
- Cross-platform claims miss the hard Windows cases: UNC paths, `MAX_PATH`/long-path policy, case-fold collisions, locked-file rename failures, and `.cmd`/PowerShell process semantics. PAL says “atomicWrite tmp + rename” but not what happens when rename is not atomic in practice. Refs: `PLATFORM_LAYER.md §4-6, §10-11`, `ARCHITECTURE.md §15`, `SAFETY_MODEL.md §14`.
- Project Brain correction heuristics are mechanically thin. A 60/180-day decay rule does not account for branch changes, mass refactors, rebases, or repo moves, so stale facts can remain “fresh” while being wrong for the active tree. Refs: `ARCHITECTURE.md §7.4`, `BOOTSTRAP_PROTOCOL.md §7`, `WORKFLOWS_SPEC.md §6`.
- Memory poisoning controls are weaker than the memory write path. Bootstrap facts, semantic promotions, and corrective signals can all be induced by untrusted repo content, but there is no quarantine tier before those facts enter default context. Refs: `BOOTSTRAP_PROTOCOL.md §6-8`, `ARCHITECTURE.md §7.1, §7.4`, `SAFETY_MODEL.md §11b`, `DEBATE_PROTOCOL.md §12`.

## What could catastrophically fail in production (3–6 bullets)
- A malicious or buggy adapter plugin exfiltrates repo code and local secrets directly via Node APIs, bypassing the orchestrator completely, while the UI still implies “trust state” exists. Refs: `SAFETY_MODEL.md §11`, `PLATFORM_LAYER.md §14`.
- Replay/audit integrity fails after an unclean shutdown or Windows file-lock race, making the “brain is the unit of replay” promise false on exactly the runs users most need to inspect. Refs: `ARCHITECTURE.md §10`, `SAFETY_MODEL.md §7`, `PLATFORM_LAYER.md §6`.
- Recursive debate/tool workflows blow through budget because retries, arbitration repair attempts, and partial-failure fallbacks all add hidden calls. The user sees “cost-first,” but the runtime still fans out unpredictably. Refs: `DEBATE_PROTOCOL.md §6, §9, §11.1`, `ARCHITECTURE.md §11`.
- Semantic memory contamination becomes self-reinforcing: poisoned facts get promoted, re-packed, cited by arbiters, then re-corroborated by later workflows until the brain becomes confidently wrong. Refs: `ARCHITECTURE.md §7.1-7.4`, `DEBATE_PROTOCOL.md §12`, `EVAL_SPEC.md §8`.

## Silent complexity bombs hiding in the architecture (4–8 bullets, NEW issues not in prior reviews)
- The docs conflict on workflow/protocol authoring format: TS modules in `ARCHITECTURE.md §13` and `DEBATE_PROTOCOL.md §13`, YAML-only MVP in `WORKFLOWS_SPEC.md §15`, Phase 4 YAML in `MVP_ROADMAP.md Phase 4`. That ambiguity will leak straight into loader design.
- `better-sqlite3` plus npm plugin ecosystem plus cross-platform prebuilt policy is a packaging trap. One native dependency is manageable; many optional native plugins are not. Refs: `ARCHITECTURE.md §9`, `PLATFORM_LAYER.md §14`, `ADAPTER_SPEC.md §11`.
- `AbortSignal` is treated as if it cancels provider work promptly, but many SDKs only cancel local awaiters. That means hidden in-flight cost and out-of-order completion against a “cancelled” workflow. Refs: `ADAPTER_SPEC.md §2, §13`, `DEBATE_PROTOCOL.md §9-10`.
- Hashing “raw provider payloads” is underspecified across SDK versions. If upstream SDK response objects reorder fields or include transient metadata, audit hashes churn for no product reason. Refs: `ADAPTER_SPEC.md §2, §3`, `DEBATE_PROTOCOL.md §7.1`.
- Bootstrap indexing treats manifests and recent git history as facts, but commit messages are user-controlled text and can become injection/poisoning carriers long before any LLM call. Refs: `BOOTSTRAP_PROTOCOL.md §4.1, §6`, `SAFETY_MODEL.md §11b`.
- Debate round parallelism needs stable transcript ordering rules. `Promise.all` completion order is not a persistence order. Without explicit canonical participant ordering, “byte-identical transcript” is fragile. Refs: `DEBATE_PROTOCOL.md §5.2, §7, §10`.

## What is unexpectedly strong (2–4 bullets)
- The docs are unusually honest about bounded determinism and plugin trust limits. That will save bad architectural decisions later. Refs: `WORKFLOWS_SPEC.md §11`, `SAFETY_MODEL.md §11`, `ARCHITECTURE.md §17.12, §17.14`.
- PAL as an enforced seam is the right call for a Node CLI targeting Windows seriously; the lint rule is a real implementation lever, not just principle. Refs: `PLATFORM_LAYER.md §1, §2b, §16`.
- Cost is treated as architecture, not reporting. That is the correct instinct for a multi-provider runtime. Refs: `ARCHITECTURE.md §11`, `OBSERVABILITY.md §3.1`.

## What should absolutely NOT be added before MVP (3–6 bullets — risks of premature complexity)
- TS-authored user workflows/protocols. A runtime code loader before isolation hardening is asking for RCE and determinism drift.
- Vector memory or embeddings. It will mask unresolved determinism and poisoning problems instead of solving them.
- Background daemons/watchers. You do not want lifecycle, lock, and crash-recovery complexity before the single-process substrate is trustworthy.
- More plugin surface than first-party adapters. The trust model is not ready.
- Team/shared brain sync. It will amplify every consistency and poisoning issue already present locally.

## What can safely be postponed until Phase 3+ (3–6 bullets)
- Provider health dashboards and routing calibration UX beyond minimal logs. Core correctness matters first.
- User-defined workflows/protocols for third parties.
- Postgres swap and any multi-workspace story.
- Rich `manthan brain` introspection beyond read-only essentials.
- Sandboxing and adapter isolation can wait structurally, but only if MVP is explicit that third-party plugins are effectively full-trust code execution.
